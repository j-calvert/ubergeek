Overview:

This package is an implementation of the Paxos Distributed Consensus algorithm, written to be as close as possible to the description provided by Appendix A1 of Leslie Lamport's "Part-time Parliament".

In particular

  • The persisted data used in the implementation (Ledger and slip of Paper) have members corresponding to data elements described in the Appendix.

  • The methods of the central classes Member and Member.BallotProcessor correspond to the (receive/send (resp.)) messages described in the Appendix.

Libraries used:

Hadoop for IPC, Berkeley DB JE for stable atomic disk storage.

Demonstration:

To run the demonstration, runAssembly.sh in the directory parent to the directory this document is in. This will start the assembly running with 5 members as separate thread groups with independent storage. Each member has a BallotProcessor thread, and IPC server threads. Then, run issueDecree.sh as many times as you like. Each time issueDecree.sh is run, a client asks a pre-specified member to initiate a new decree (with some dummy data). Eg.

> ~/school/paxos$ ./runAssembly.sh &

(either run the above in the background with the “&” or open a new shell)

(wait a few seconds for the assembly to finish initializing)
> ~/school/paxos$ ./issueDecree.sh
> ~/school/paxos$ ./issueDecree.sh

....

Feasibility (aka, why I think it works):

I believe that the algorithm itself works because it's been well established as a reliable means of achieving distributed consensus. This implementation is not meant to verify this.

I claim that the implementation is true to the prescribed algorithm because the individual Member (and Member.BallotProcessor) methods are small and correspond directly to actions described in Appendix A1. I've tested each of these methods by stepping through them in debug modes with a variety of different arguments.

I do not provide means to simulate member failures in the demonstration. I did some preliminary tests of the durability of the implementation in the event of member failure by running the assembly inside an SDK. I ran it in debug mode and inserted breakpoints in receive (Member) methods. By stopping the execution at these points, I simulated temporary failure of the member in question. After resuming execution, subsequent ballots for affected decrees succeeded.

Limitations:

  • The set of members in the assembly is assumed to be constant, in that new members do not enter the assembly and existing members do not (permanently) leave the assembly.

  • There is no distinguished proposer, but rather, the proposer is chosen by the client issuing request for a new decree to the assembly. Hence, this implementation does not guarantee progress. In addition, it opens the possibility of a client-requested decree failing entirely. At very least, a more robust implementation would provide some kind of response (indicating success or failure) for a client-requested decree.

Objects:

I've taken care to document data elements and methods, and to model them closely to the semantics of the cited paper. These notes are available in the surrounding javadocs.

The primary classes of interest are Member, Member.BallotProcessor, Ledger, slip of Paper.

Files:

The contents of this submission are:

build.properties & build.xml

Ant scalars and build script.

data/

Directory created by runAssembly.sh script, contains one subdirectory per Sleepycat BDB environment (one per member on this server) (will not exist until you run runAssembly.sh)

doc/

This document and the javadocs for the code written for this assignment.

issueDecree.sh

Shell script to create a member client and use it to have that member issue a new decree.

lib/

External, non JDK libraries used (see section below) and a jar of the code written for this assignment.

README.txt/

A small text file pointing the reader to this document.

runAssembly.sh

Shell script to run a demonstration assembly of 5 members on this server.

src/

Well documented :-) Java source code written for this assignment
